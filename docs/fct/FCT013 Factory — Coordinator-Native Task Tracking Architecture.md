# FCT013 Factory ‚Äî Coordinator-Native Task Tracking Architecture

**Prefix:** FCT | **Repo:** `~/dev/factory/` | **Date:** 2026-03-20 | **Status:** Draft

---

## 1. Executive Summary

Factory currently maintains two disconnected task worlds. The FCT012 markdown backlog [1] serves as the canonical work item registry, while the portal reads a `tasks.json` file served by a Python GSD sidecar (`server.py` on port 41935). Neither system talks to coordinator-rs, which already owns the machinery for task checkout, budget enforcement, and event streaming.

The target state is straightforward: coordinator-rs becomes the single authority for task lifecycle. The approach is incremental ‚Äî bridge the two worlds first with a sync script, then progressively move ownership into the coordinator until the GSD sidecar and markdown backlog become archival artifacts.

This document is Layer 3 (architecture and roadmap only; no implementation).

## 2. Current State

### 2.1 Portal + GSD Sidecar

The React portal (v8, deployed at `:41988`) renders task state from `tasks.json` via the GSD sidecar (`server.py` on `:41935`). The portal `TaskRecord` interface defines the current schema:

```typescript
interface TaskRecord {
  id: string;
  title: string;
  description?: string;
  status: "pending" | "in-progress" | "done" | "blocked";
  effort?: string;
  order: number;
  blocked_by: string[];
  block: string;
  assignee?: string | null;
  created?: string;
  updated?: string;
}
```

The wrapping `TasksDocument` also carries `blocks` (category metadata), `log` (audit trail entries), and document-level timestamps. The portal dashboard supports status cycling, dependency unblocking, and log entry generation ‚Äî all operating directly on this JSON structure [2].

### 2.2 Coordinator-rs Modules

Coordinator-rs (approximately 11,600 lines Rust, 37 tests passing) already contains the building blocks for task-aware orchestration [3]:

- **task_lease.rs** ‚Äî Atomic task checkout via `TaskLeaseManager`. Each `TaskLease` carries `task_id`, `agent_id`, `claimed_at`, and a configurable `ttl` (default 5 minutes). Expired leases are automatically reclaimed. Backed by an in-memory `HashMap<String, TaskLease>` behind a `tokio::sync::Mutex`.
- **run_events.rs** ‚Äî Append-only JSONL event streaming. Each `RunEvent` carries `run_id`, `seq`, `event_type` (12 variants including `SessionStart`, `SessionEnd`, loop lifecycle events, and `FrozenHarnessViolation`), `stream`, `level`, `message`, `payload`, and `timestamp`. Events are written per-run to `runs_dir/{run_id}.jsonl`.
- **budget.rs** ‚Äî Per-agent monthly budget tracking with soft/hard thresholds and pause enforcement.
- **approval.rs** ‚Äî Typed approval gates (5 variants) with HMAC-signed decision files.

### 2.3 The Gap

There is no link between these systems. The `RunEvent` struct has no `task_id` field ‚Äî runs exist independently of tasks. The `TaskLeaseManager` operates on opaque string IDs without reading any task store. `tasks.json` is hand-edited or will be generated by a sync script (`sync-fct012.py`). FCT012 markdown is the canonical backlog, but agents cannot consume it programmatically without parsing markdown tables.

## 3. Target State

Coordinator-rs owns the full task lifecycle:

1. **Task store** ‚Äî coordinator-rs reads and writes `tasks.json` as its persistent task store.
2. **Run-to-task linking** ‚Äî every `RunEvent` carries an optional `task_id`, establishing provenance between runs and the tasks that triggered them.
3. **Portal integration** ‚Äî the portal queries coordinator-rs directly via HTTP REST endpoints (FCT-064) instead of polling the GSD sidecar.
4. **Status feedback loop** ‚Äî task status transitions are driven by dispatch outcomes:

```
dispatch claim   ->  status: in-progress
run success      ->  status: done
run failure      ->  status: blocked
lease expiry     ->  status: pending (reclaimed)
```

Budget and approval gates (already implemented) apply at the dispatch boundary, gating task execution before a lease is granted.

## 4. Paperclip Reference

Three Paperclip patterns directly inform this design [4][5]:

**Atomic checkout model (`issues.ts`)** ‚Äî Paperclip's issue checkout ensures only one agent works on a given issue at a time. This pattern is already adapted as `task_lease.rs` in coordinator-rs. The Factory adaptation uses in-memory leases with TTL-based expiry rather than Paperclip's database-backed locking, which is appropriate for single-instance deployment.

**Heartbeat and liveness (`heartbeat.ts`)** ‚Äî Paperclip uses heartbeat signals to detect stalled agents and reclaim work. In Factory, this maps to the lease TTL mechanism: a lease that is not renewed within its TTL window is automatically expired and the task becomes available for re-dispatch. No separate heartbeat protocol is needed at current scale.

**Run-to-issue linking** ‚Äî Each Paperclip run references its source issue, creating a bidirectional relationship between work items and execution history. This is the key missing piece in coordinator-rs: the `RunEvent` struct needs a `task_id` field to establish the same linkage.

## 5. Factory Adaptation

### 5.1 New Module: `task_store.rs`

A new module responsible for loading, indexing, and persisting `tasks.json`:

- Load `tasks.json` on startup, deserialize into `Vec<TaskRecord>` with an in-memory `HashMap<String, usize>` index by task ID.
- Atomic file writes via the standard pattern: serialize to `serde_json`, write to tempfile in the same directory, `rename()` over the target. This prevents partial writes from corrupting the task store.
- Expose methods: `get_task()`, `update_status()`, `list_by_status()`, `list_by_assignee()`.

### 5.2 RunEvent Extension

The `RunEvent` struct gains one field:

```rust
pub struct RunEvent {
    // ... existing fields ...
    pub task_id: Option<String>,  // NEW: links run to source task
}
```

This is an additive change. Existing JSONL logs remain valid ‚Äî `task_id` deserializes as `None` for historical events.

### 5.3 Status Feedback

On dispatch claim (lease granted): `task_store.update_status(task_id, "in-progress")`.
On run success: `task_store.update_status(task_id, "done")` and call `unblock_dependents()`.
On run failure: `task_store.update_status(task_id, "blocked")`.
On lease expiry (agent crash/timeout): `task_store.update_status(task_id, "pending")`.

### 5.4 Budget and Approval Integration

Existing `approval.rs` gates apply at the task dispatch boundary. Before granting a lease, the coordinator checks:

1. Agent budget has headroom (`budget.rs` soft/hard threshold check).
2. No pending approval gate blocks the task's category or agent.
3. Task dependencies (`blocked_by`) are all in `done` status.

### 5.5 REST Endpoints (Part of FCT-064)

Task-specific endpoints for portal consumption:

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/tasks` | List all tasks, optional `?status=` and `?assignee=` filters |
| `PUT` | `/tasks/{id}/status` | Update task status (operator override) |
| `GET` | `/tasks/{id}/runs` | List RunEvents linked to a task |
| `GET` | `/tasks/{id}` | Single task detail with linked run summary |

## 6. Data Model

### 6.1 Current Schema (portal `types.ts`)

```json
{
  "id": "FCT-055",
  "title": "Wire coordinator-rs modules into main dispatch loop",
  "description": "...",
  "status": "pending | in-progress | done | blocked",
  "effort": "l",
  "order": 1,
  "blocked_by": [],
  "block": "infrastructure",
  "assignee": "boot",
  "created": "2026-03-20T00:00:00Z",
  "updated": "2026-03-20T00:00:00Z"
}
```

### 6.2 Target Additions (Additive Only)

No existing fields are modified or removed. Four optional fields are added:

```json
{
  "...current fields...",
  "run_ids": ["run_abc123", "run_def456"],
  "last_run_status": "success | failure | timeout",
  "claimed_by": "boot",
  "claimed_at": "2026-03-20T14:30:00Z"
}
```

- `run_ids` ‚Äî ordered list of run IDs that have been dispatched for this task. Populated by coordinator-rs on dispatch.
- `last_run_status` ‚Äî outcome of the most recent run. Enables portal to show run health without querying the full JSONL log.
- `claimed_by` ‚Äî agent currently holding the lease. `null` when unclaimed.
- `claimed_at` ‚Äî ISO 8601 timestamp of the current lease grant. `null` when unclaimed.

The portal `TaskRecord` TypeScript interface gains matching optional fields. Existing portal code continues to function ‚Äî all additions are optional.

## 7. Migration Path

### Phase 0 ‚Äî Bridge (Current)

A Python sync script (`sync-fct012.py`) parses the FCT012 markdown tables and generates `tasks.json`. Execution is manual or cron-triggered. The GSD sidecar serves the resulting file to the portal. Coordinator-rs is unaware of tasks.

**Exit criterion:** `sync-fct012.py` produces valid `tasks.json` that the portal renders correctly.

### Phase 1 ‚Äî Read-Only Integration (Post FCT-055)

Coordinator-rs loads `tasks.json` via `task_store.rs` on startup. The `RunEvent` struct gains the `task_id` field. When dispatching work, the coordinator references the task store to select the next eligible task and records the `task_id` on all emitted run events.

The coordinator does not write back to `tasks.json` in this phase. The sync script remains the source of truth for task content. The GSD sidecar continues serving the portal.

**Exit criterion:** Run JSONL logs contain `task_id` fields. `TaskLeaseManager` claims are correlated with task store entries.

### Phase 2 ‚Äî Write Integration (Post FCT-064)

Coordinator-rs writes task status updates back to `tasks.json` via the atomic file write pattern. REST endpoints serve task state directly to the portal. The GSD sidecar becomes optional ‚Äî the portal can be reconfigured to query coordinator-rs instead.

**Exit criterion:** Portal reads task state from coordinator-rs HTTP API. Task status transitions are driven by dispatch outcomes. GSD sidecar can be stopped without portal degradation.

### Phase 3 ‚Äî Full Authority (Post GSD Deprecation)

Coordinator-rs is the sole task authority. The GSD sidecar (`server.py`) is deprecated per FCT-040 [1]. FCT012 markdown becomes an archival reference ‚Äî new tasks are created through the coordinator or its API. The sync script is retired.

**Exit criterion:** `server.py` removed from systemd. Portal exclusively queries coordinator-rs. FCT012 receives a final status update marking it as archival.

## 9. Open Questions

1. **File watch vs. SIGHUP** ‚Äî Should coordinator-rs watch `tasks.json` for external changes (e.g., sync script updates), or require an explicit reload signal? File watching adds a dependency (`notify` crate) and introduces race conditions with concurrent writes. SIGHUP is simpler but requires manual triggering. A third option: reload on a configurable interval (e.g., every 60 seconds) with file mtime check.

2. **Completed item retention** ‚Äî Should coordinator-rs load and track completed (`done`) items, or filter them out at load time? Completed items are useful for dependency resolution (`blocked_by` a done task = unblocked) but accumulate over time. Recommendation: load all items, but only serve active items via REST unless `?include_done=true` is specified.

3. **Cost attribution** ‚Äî How should run cost be attributed back to tasks when MLX-LM lacks per-request token metering? The current `budget.rs` uses invocation count as a proxy [3][6]. Options: (a) attribute invocation count to task, (b) estimate tokens from output length, (c) defer until MLX-LM exposes metering (FCT-068).

4. **Conflict resolution** ‚Äî If both the sync script and coordinator-rs write to `tasks.json`, who wins? In Phase 1 (read-only), no conflict exists. In Phase 2, the coordinator owns status fields while the sync script owns content fields (title, description, effort). A field-level merge strategy or write lock protocol is needed. Simplest resolution: once Phase 2 begins, the sync script writes only new tasks (append-only) and the coordinator owns all status mutations.

5. **Task creation** ‚Äî Should coordinator-rs be able to create new tasks, or only consume what `sync-fct012.py` produces? Creating tasks from within the coordinator enables agent-initiated work items (e.g., Boot discovers a bug and creates a task). However, this introduces a second source of truth for task content. Deferring task creation to Phase 3 keeps the model simple during the bridge period.
