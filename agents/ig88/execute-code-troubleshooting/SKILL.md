---\nname: execute-code-troubleshooting\ndescription: Troubleshooting approach for when the execute_code tool fails to properly execute complex Python scripts\ncategory: software-development\n---\n\n# Execute Code Tool Troubleshooting\n\nWhen the `execute_code` tool returns empty output or behaves unexpectedly with complex Python scripts, use this systematic approach to diagnose and work around the issue.\n\n## Symptoms\n- Tool returns `{\"status\": \"success\", \"output\": \"\", \"tool_calls_made\": 0, ...}`\n- Simple commands like `print(\"hello\")` work, but complex scripts fail\n- No error messages are returned, just empty output\n- The tool appears to accept the code but not execute it properly\n\n## Diagnostic Approach\n\n### 1. Test Basic Functionality\nStart with the simplest possible code to verify the tool works at all:\n```python\nprint(\"hello\")\n```\n\nIf this works, the basic tool functionality is intact.\n\n### 2. Test Module Imports\nIf your script uses external modules, test them individually:\n```python\nimport requests\nprint(requests.__version__)\n```\n\nThis confirms whether the module is available in the execution environment.\n\n### 3. Test HTTP Requests Separately\nIf your script makes HTTP calls, test them in isolation:\n```python\nimport requests\nr = requests.get('https://httpbin.org/get')\nprint(r.status_code)\n```\n\n### 4. Test Data Processing Steps Separately\nBreak down complex operations into smaller chunks:\n```python\nimport requests\nr = requests.get('https://api.binance.com/api/v3/ticker/price?symbol=BTCUSDT')\nprint(r.status_code)\ndata = r.json()\nprint(type(data))\nprint(data.keys() if isinstance(data, dict) else 'not a dict')\n```\n\n### 5. Check for Silent Failures\nThe tool may be failing silently at certain operations. Test each step:\n- Variable assignments\n- JSON parsing\n- Mathematical operations\n- Conditional statements\n\n## Common Issues and Workarounds\n\n### Issue: Empty Output with Complex Scripts\n**Symptom**: Tool accepts code but returns empty output\n**Workaround**: \n- Reduce script complexity incrementally\n- Add explicit print statements after each major step\n- Avoid chaining operations; break into separate steps\n- Use intermediate variables and print them\n\n### Issue: Module Not Available\n**Symptom**: ImportError or ModuleNotFoundError (though sometimes not reported)\n**Workaround**:\n- Test the import first with a simple version\n- Check if the module is in the ig88 venv (should be available)\n- Consider if the tool has restrictions on certain modules\n\n### Issue: HTTP Requests Fail Silently\n**Symptom**: Requests appear to succeed but don't return expected data\n**Workaround**:\n- Always check status_code explicitly\n- Print response headers and a sample of content\n- Verify you're getting JSON when expected\n- Consider timeout and connection issues\n\n## Best Practices for Reliable Execution\n\n1. **Start Simple**: Begin with the most basic version of your script\n2. **Add Increments**: Add one piece of functionality at a time\n3. **Validate Each Step**: Print intermediate results after each operation\n4. **Explicit Returns**: Make sure your script prints or returns something visible\n5. **Isolate External Calls**: Test API calls separately from data processing\n6. **Check Types**: Verify data types at each step (especially after JSON parsing)\n7. **Avoid Complex One-Liners**: Break `data['key']['subkey']` into multiple steps\n\n## Example Regime Detection Script (Working Approach)\n\nInstead of running the entire regime detection script at once, break it down:\n\n```python\n# Step 1: Test BTC price fetch\nimport requests\nbtc_r = requests.get('https://api.binance.com/api/v3/ticker/price?symbol=BTCUSDT')\nprint(f\"BTC status: {btc_r.status_code}\")\nbtc_data = btc_r.json()\nprint(f\"BTC data type: {type(btc_data)}\")\nbtc_price = float(btc_data['price'])\nprint(f\"BTC price: {btc_price}\")\n\n# Step 2: Test Fear & Greed index\nfg_r = requests.get('https://api.alternative.me/fng/?limit=1')\nprint(f\"FG status: {fg_r.status_code}\")\nfg_data = fg_r.json()\nprint(f\"FG data: {fg_data}\")\nfg_value = int(fg_data['data'][0]['value'])\nprint(f\"Fear & Greed value: {fg_value}\")\n\n# Step 3: Calculate regime\nscore = 5  # neutral baseline\nif fg_value > 60: score += 1\nelif fg_value < 30: score -= 1\n\nstate = 'RISK_ON' if score >= 7 else 'RISK_OFF' if score <= 3 else 'NEUTRAL'\nprint(f\"Regime score: {score}, state: {state}\")\n\n# Step 4: Output final result\nfrom datetime import datetime, timezone\nresult = {\n    'timestamp': datetime.now(timezone.utc).isoformat(),\n    'regime': {'state': state, 'score': score, 'confidence': 0.6},\n    'btc_price': btc_price,\n    'fear_greed': fg_value\n}\nprint(f\"Final result: {result}\")\n```\n\n## Verification\nAfter implementing your workaround, verify:\n- Each step produces expected output\n- The final result matches what you expect\n- You can reconstruct the full script from working components\n\n## Prevention\nFor future scripts:\n- Design them to be broken into testable chunks from the start\n- Include validation checkpoints\n- Consider writing more complex scripts to files and executing them via terminal tool if execute_code proves unreliable\n- Keep a library of known-working snippets for common operations (API calls, JSON parsing, etc.)
